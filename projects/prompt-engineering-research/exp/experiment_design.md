# CAPE 实验设计文档

## 1. 研究概述

### 1.1 研究目标

本研究旨在开发并验证**基于任务复杂度的自适应 Prompt 工程框架（Complexity-Adaptive Prompt Engineering, CAPE）**，通过动态选择最优 Prompt 策略，在保持代码生成质量的同时显著降低计算成本。

### 1.2 核心研究问题

1. **RQ1**：是否存在一个最优的 Prompt 策略选择函数，能够根据任务复杂度最大化质量 - 成本比（Q/C）？
2. **RQ2**：代码任务的多维度复杂度特征能否有效预测最优 Prompt 策略？
3. **RQ3**：引入动态调整机制（基于生成中间结果的质量反馈）是否能进一步提升 Q/C 比？
4. **RQ4**：最优复杂度阈值在不同模型家族间是否存在系统性差异？复杂度特征的重要性排序是否具有跨模型一致性？

### 1.3 研究假设

| 假设编号 | 假设内容 | 验证方法 |
|---------|---------|---------|
| **H1** | 存在最优 Prompt 策略选择函数 f: C → S 最大化 Q/C 比 | 对比 CAPE 与静态策略的 Q/C 比 |
| **H1.1** | 代码任务复杂度可通过多维度特征有效预测 | 训练复杂度预测器，评估 R² |
| **H1.2** | 存在复杂度阈值 T₁ 和 T₂ 划分最优策略区间 | 贝叶斯优化搜索阈值，验证 Q/C 提升 |
| **H1.3** | 动态调整机制可提升 Q/C 比 15% 以上 | 对比启用/禁用动态调整器的性能 |
| **H1.4** | 复杂度特征重要性排序具有跨模型一致性 | 计算跨模型特征重要性 Spearman 相关系数 |

## 2. 实验设计

### 2.1 实验类型

本研究采用**混合实验设计**，结合：
- **对照实验**：CAPE 框架 vs 静态策略基线
- **消融实验**：验证各组件（复杂度预测、阈值决策、动态调整）的贡献
- **跨模型验证**：在 8 个不同模型上验证泛化能力

### 2.2 实验变量

#### 自变量（Independent Variables）

| 变量名称 | 类型 | 取值 |
|---------|------|------|
| **Prompt 策略** | 分类变量 | Zero-shot, Few-shot, CoT |
| **任务复杂度** | 连续变量 | [0, 1] 复杂度得分 |
| **模型类型** | 分类变量 | GPT-4o, CodeLlama-34B, Claude-3.5, GPT-3.5, GPT-4, Llama-3-70B, Mistral, Gemini |
| **动态调整** | 二元变量 | 启用, 禁用 |

#### 因变量（Dependent Variables）

| 变量名称 | 类型 | 测量方法 |
|---------|------|---------|
| **pass@1** | 连续变量 | 一次性通过测试的比例 |
| **pass@10** | 连续变量 | 10 次采样至少一次通过的比例 |
| **token 消耗** | 连续变量 | 输入 + 输出 token 总数 |
| **Q/C 比值** | 连续变量 | pass@1 / 平均 token 消耗 |
| **代码可执行率** | 连续变量 | 无语法错误代码的比例 |

#### 控制变量（Control Variables）

- 温度参数（temperature = 0.7）
- 最大 token 数（max_tokens = 2048）
- 测试用例集（固定测试集）
- 示例数量（Few-shot 固定为 5 个示例）

### 2.3 实验组设计

| 实验组 | 描述 | 目的 |
|-------|------|------|
| **Static-ZeroShot** | 统一使用 Zero-shot 策略 | 基线对比 |
| **Static-FewShot** | 统一使用 Few-shot 策略 | 基线对比 |
| **Static-CoT** | 统一使用 CoT 策略 | 主要基线对比 |
| **PET-Select** | 复现 Wang et al. (2024) 二分类方法 | 与现有方法对比 |
| **RoutingGen** | 复现 Li et al. (2025) 动态路由方法 | 与现有方法对比 |
| **CAPE-Static** | CAPE 框架（禁用动态调整） | 验证阈值决策效果 |
| **CAPE-Full** | CAPE 框架（完整功能） | 验证整体框架效果 |

### 3.3 复杂度特征提取

#### 结构复杂度特征
- **循环嵌套深度**：代码中循环结构的最大嵌套层数
- **条件分支数**：if-else、switch-case 等条件语句数量
- **函数调用层数**：函数调用的最大深度

#### 语义复杂度特征
- **算法类型**：排序、搜索、动态规划、图论、树操作等（one-hot 编码）
- **数据结构使用**：列表、字典、集合、自定义类等

#### 依赖复杂度特征
- **API 调用数量**：标准库和第三方库 API 调用次数
- **第三方库依赖数**：所需第三方库的数量

#### 文本复杂度特征
- **描述长度**：自然语言描述的 token 数
- **歧义性评分**：基于语言模型的不确定性评分

## 4. 评估指标

### 4.1 质量指标

#### pass@k
$$\text{pass@k} = \mathbb{E}_{\text{problems}} \left[ 1 - \frac{\binom{n-c}{k}}{\binom{n}{k}} \right]$$

其中：
- $n$ = 总采样数（本研究使用 n=10）
- $c$ = 通过所有测试用例的采样数
- $k$ = 评估的采样数（主要报告 pass@1 和 pass@10）

**目标**：CAPE 框架 pass@1 不低于静态 CoT 策略的 95%

#### 代码可执行率
$$\text{Executability} = \frac{\text{可执行代码数}}{\text{总生成代码数}}$$

**目标**：> 90%

### 4.2 成本指标

#### Token 消耗
$$\text{Token}_{\text{total}} = \sum_{i=1}^{N} (\text{input\_tokens}_i + \text{output\_tokens}_i)$$

**目标**：CAPE 框架相比静态 CoT 节省 40-60% token

#### API 调用次数
$$\text{API}_{\text{avg}} = \frac{\text{总 API 调用次数}}{\text{任务数}}$$

**目标**：CAPE 框架平均 API 调用次数 < 1.5

### 4.3 效率指标

#### 质量 - 成本比（Q/C）
$$\text{Q/C} = \frac{\text{pass@1}}{\text{average\_token\_consumption}}$$

**目标**：CAPE 框架 Q/C 比静态 CoT 提升 50% 以上

#### 动态调整提升率
$$\text{Improvement} = \frac{\text{Q/C}_{\text{with\_adjustment}} - \text{Q/C}_{\text{without\_adjustment}}}{\text{Q/C}_{\text{without\_adjustment}}} \times 100\%$$

**目标**：提升 ≥ 15%（验证假设 H1.3）

### 4.4 统计指标

#### 配对 t 检验
用于检验 CAPE 与基线方法的性能差异是否显著：
- 零假设 H₀：CAPE 与基线方法性能无显著差异
- 备择假设 H₁：CAPE 性能显著优于基线方法
- 显著性水平：α = 0.05

#### 效应量（Cohen's d）
$$d = \frac{\bar{X}_1 - \bar{X}_2}{s_{\text{pooled}}}$$

其中 $s_{\text{pooled}} = \sqrt{\frac{(n_1-1)s_1^2 + (n_2-1)s_2^2}{n_1+n_2-2}}$

**目标**：d > 0.5（中等效应）

通过对比各消融组的 Q/C 比，量化各组件的贡献：
- **复杂度预测贡献** = Ablation-2 Q/C - Ablation-1 Q/C
- **阈值决策贡献** = Ablation-3 Q/C - Ablation-2 Q/C
- **动态调整贡献** = Ablation-4 Q/C - Ablation-3 Q/C

### 7.1 主要对比检验

| CAPE-Full vs RoutingGen | 配对 t 检验 | α = 0.05 | p < 0.05, d > 0.5 |
| CAPE-Full vs CAPE-Static | 配对 t 检验 | α = 0.05 | p < 0.05（验证 H1.3） |

### 7.2 多重检验校正

由于进行多次统计检验，使用**Bonferroni 校正**控制家族错误率：
- 校正后显著性水平：α' = α / n（n = 检验次数）

    max_loop_depth = 0

        'branch_count': branch_count,
